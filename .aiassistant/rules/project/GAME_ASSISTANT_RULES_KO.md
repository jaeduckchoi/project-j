# 종구의 식당 AI 작업 규칙

## 1. 프로젝트 개요

- 이 저장소는 Unity 기반 2D 탑다운 탐험 + 식당 운영 프로토타입 프로젝트다.
- 프로젝트명은 `종구의 식당`이다.
- 핵심 루프는 `허브 상태 확인 -> 오전 탐험 -> 재료 수집 -> 허브 복귀 -> 메뉴 선택 -> 간단한 영업 -> 정산 -> 성장 -> 다음 날 전환`이다.
- 전체 체감 비중은 `탐험 8 : 장사 2`를 기준으로 잡고, 장사는 복잡한 실시간 경영보다 결과 처리 중심으로 단순화한다.
- 기본 응답 언어는 한국어를 우선한다.

## 3. 프로토타입 범위 기준

- 플레이어 조작은 `WASD` 또는 방향키 이동 + `E` 상호작용을 기본으로 하고, 클릭 이동은 사용하지 않는다.
- 미니맵은 기본 범위에서 제외하고, 대신 지역 이름 표시, 짧은 안내 문구, 최소한의 방향 안내로 탐험 흐름을 돕는다.
- 별도 긴 튜토리얼 스테이지보다 허브 진입, 지역 입장, 첫 채집, 첫 장사 시점의 짧은 설명문으로 안내를 녹여 넣는다.
- 탐험 스킵과 장사 스킵은 프로토타입 반복 테스트용 기능으로 유지한다.
- 허브는 단순 메뉴 화면이 아니라 직접 걸어 다니며 `주방`, `장부/게시판`, `창고`, `작업대`, `출입문`을 상호작용하는 안전한 공간으로 유지한다.

## 4. 게임 설계 기준

- 인벤토리는 채집 자원 전용 슬롯 구조를 유지하고, 시작 `8칸`에서 업그레이드로 `12칸`, `16칸`까지 확장한다.
- 도구는 인벤토리를 차지하지 않는 상시 해금형 구조로 유지하고, 기본 기준은 `갈퀴`, `낚시대`, `낫`, `랜턴`이다.
- 도구 슬롯, 탐험 중 장착 교체, 버리기 같은 시스템은 범위를 넓히지 않는 이상 기본 구현 대상으로 두지 않는다.
- 창고는 허브에서 여는 단순 보관함으로 유지하고, 핵심 기능은 `맡기기 / 꺼내기`다.
- 업그레이드는 `골드 + 특정 재료`를 함께 소모하는 데이터 기반 구조를 유지한다.
- 장사는 메뉴 선택, 필요 재료 확인, 결과 정산, 골드 / 평판 반영 중심으로 단순하게 유지한다.
- 탐험 지역은 `바닷가 -> 깊은 숲 -> 폐광산 -> 바람 언덕` 확장 순서를 기본으로 본다.
- 폐광산은 랜턴이나 시야 제약 같은 진입 조건을, 바람 언덕은 강풍과 숏컷 같은 지역 고유 기믹을 유지한다.

## 5. 구현 구조 기준

- 핵심 시스템은 `PlayerController`, `InteractionDetector`, `IInteractable`, `InventoryManager`, `StorageManager`, `RestaurantManager`, `EconomyManager`, `UpgradeManager`, `ScenePortal`, `DayCycleManager/FlowManager`, `UIManager`처럼 역할을 분리해 유지한다.
- 자원, 레시피, 업그레이드, 지역 해금 조건은 가능한 한 ScriptableObject 또는 데이터 중심 구조로 관리한다.
- 하드코딩을 줄이고, 초보 개발자도 읽기 쉬운 변수명 / 메서드명 / null 체크를 유지한다.
- Unity 직렬화 파일과 에셋 참조는 영향 범위가 크므로 참조 경로까지 함께 수정한다.
- generated YAML이나 씬 데이터가 빌더 코드의 결과물이라면, 결과물만 임시 수정하지 말고 생성 경로를 먼저 정리한다.
- UI를 바꾸면 `Assets/Scripts/UI/UIManager.cs`와 `Assets/Editor/JongguMinimalPrototypeBuilder.cs`를 함께 확인한다.
- 새 런타임/에디터 스크립트는 폴더 경로 기준 네임스페이스를 맞춘다.
- 폴더명이 `Camera`, `Editor`처럼 Unity/.NET 주요 타입과 충돌하면 `GameCamera`, `ProjectEditor`처럼 충돌 없는 예외 네임스페이스를 사용하고, 예외 이유가 반복되면 규칙 문서에도 반영한다.
- 기존 `MonoBehaviour`, `ScriptableObject`, 직렬화 가능한 타입을 네임스페이스로 이동할 때는 `UnityEngine.Scripting.APIUpdating.MovedFrom`으로 이전 경로를 보존한다.
- 플레이어 비주얼 크기나 방향 스프라이트를 바꿀 때는 물리 루트와 비주얼 루트의 책임을 구분한다.
- 사용자가 만든 기존 변경은 명시적 요청 없이 되돌리지 않는다.

## 6. UI / 텍스트 규칙

- TextMesh Pro 폰트와 기본 TMP 설정을 변경할 때는 실제 참조 에셋 경로까지 확인한다.
- UI 텍스트는 겹침, 잘림, 대비, 패널 밖 overflow를 함께 점검한다.
- 허브 팝업 UI(`요리 메뉴`, `업그레이드`, `재료`, `창고`)가 열리면 게임 진행을 일시 정지하고, 닫히면 원래 시간 흐름을 복구한다.
- 라벨, 프롬프트, 가이드, 결과 텍스트 위치는 빌더와 런타임 보정 코드가 같은 기준을 쓰도록 유지한다.
- 허브 팝업에서 씬에 직접 지정한 `Image.sprite`, `PopupTitle`, `PopupLeftCaption`의 폰트/배치 값은 명시적 요청 없이는 초기화하거나 다른 기준으로 덮어쓰지 않는다.
- 대체된 버튼, 레거시 카드, 오래된 도크, 사용하지 않는 중복 UI 경로는 남기지 않는다.

## 7. 주석과 문서 규칙

- 메서드 단위 설명과 복잡한 로직 앞의 블록 주석은 유지한다.
- 새로 추가하거나 수정하는 코드에는 현재 동작 기준이 드러나도록 메서드 설명과 복잡한 블록 주석을 한글로 유지한다.
- 작업 중 닿은 파일에 무주석 핵심 메서드나 블록이 보이면 같은 변경에서 함께 보강한다.
- 단순 대입이나 자명한 동작을 설명하는 장황한 주석은 늘리지 않는다.
- 새로 추가하거나 수정하는 주석과 문서는 UTF-8 한글 기준으로 작성하고, 기존 영어 주석을 갱신할 때도 의미를 유지한 한글 주석으로 통일한다.
- 기능을 바꾸면 관련 문서도 함께 갱신한다.
- 문서는 추상 설명보다 실제 게임 기준과 현재 구현 상태를 기준으로 적는다.
- 규칙 문서에 새 기준이 생기면 `AGENTS.md`와 project rules 문서에서 모두 보이게 연결한다.

## 8. 폰트와 에셋 네이밍 규칙

- `Assets/Generated/Fonts` 아래 생성 폰트 에셋과 원본 폰트 파일명은 lower camelCase를 사용한다.
- 파일명을 바꾸면 빌더 코드, 문서, TMP 참조 경로를 함께 갱신한다.
- 가급적 파일명 규칙과 빌더가 생성하는 asset 이름 규칙을 일치시킨다.

## 9. 검증 규칙

- Unity 플레이 테스트나 컴파일을 직접 확인하지 못했으면 그 사실을 명시한다.
- 자동 감사나 배치 컴파일이 있으면 결과를 함께 확인한다.
- 실행 검증이 불가능하면 어떤 파일과 어떤 좌표 또는 어떤 참조를 기준으로 검토했는지 구체적으로 남긴다.
- 생성 구조나 네임스페이스를 바꾸면 저장된 씬, 관련 `using`, 빌더, 자동 감사 코드, 배치 컴파일 결과가 같은 기준을 보도록 함께 정리한다.
## 10. Git 커밋 메시지 규칙

- 모든 Git 커밋 메시지는 한글로 작성한다.
- 기본 형식은 `type : subject`이며, 필요할 때만 제목 아래 한 줄을 비우고 `body`, `footer`를 이어서 작성한다.
- 제목은 50자 이내로 작성하고, 제목 끝에 마침표를 붙이지 않는다.
- `type`은 아래 사용 가능한 소문자 목록만 사용한다.
- 영어 diff 요약, PR 제목, 자동 생성 커밋 초안이 입력으로 주어져도 최종 커밋 메시지는 자연스러운 한글로 다시 작성한다.
- 파일 경로, 코드 식별자, 브랜치명처럼 번역하면 안 되는 고유 명칭을 제외하면 영문 문장을 제목이나 본문에 그대로 쓰지 않는다.
- 제목만으로 변경 이유와 핵심이 충분히 드러나면 본문과 footer는 생략한다.
- 본문은 제목 아래 한 줄을 비우고 작성한다.
- 본문은 왜 바꿨는지와 무엇을 바꿨는지를 짧고 구체적으로 적는다.
- footer는 이슈 번호, 후속 작업, 브레이킹 변경처럼 추가 맥락이 필요할 때만 작성한다.
- squash merge 커밋 메시지는 `[squash] 브랜치명` 형식을 사용한다. 예: `[squash] hotfix/blabla`
- Git 로컬 `commit.template`은 `.aiassistant/rules/project/GIT_COMMIT_TEMPLATE_KO.md`를 기준으로 사용하고, 템플릿 내용을 바꾸면 이 섹션과 함께 갱신한다.

### 사용 가능한 type

- `feat` : 새로운 기능 추가
- `update` : 기존 기능 수정
- `fix` : 버그 수정
- `docs` : 문서 및 주석 수정
- `design` : CSS 등 UI 디자인 변경
- `style` : 오타, 코드 포맷팅, 세미콜론, 띄어쓰기 등 동작 변화가 없는 수정
- `rename` : 파일명 수정
- `delete` : 필요 없는 파일 삭제
- `refactor` : 코드 리팩토링
- `test` : 테스트 코드 추가
- `chore` : 빌드 설정, 프로젝트 설정, import 수정, 함수명 수정 등

### 작성 형식

```text
type : subject

body

footer
```

### 예시

```text
feat : 사용자 정보 가져오기

로그인한 사용자의 정보를 JSON에 담아 가져온다
```

```text
fix : 이미지 경로 수정

이미지 경로에 포함된 불필요한 경로를 제거하도록 수정한다
```

```text
docs : 커밋 메시지 규칙 정리
```

```text
fix : 허브 팝업 닫기 버튼 스프라이트 복구

빌더 재생성 시 닫기 버튼 이미지가 None 으로 초기화되던 경로를 막았다.
팝업 스프라이트 시트 매핑을 씬 기준으로 정리했다.

관련: popup ui sync
```
```text
design : 깊은 숲 씬 UI 요소와 레이아웃 보강

깊은 숲 씬에 안내 오브젝트와 이동 구역 설정을 추가하고
UI 배치와 텍스트 정렬, 자동 크기 조정을 함께 정리했다
```
